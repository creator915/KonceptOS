# KonceptOS

**形式概念分析作为开发操作系统。**

KonceptOS 将[形式概念分析（Formal Concept Analysis）](https://en.wikipedia.org/wiki/Formal_concept_analysis)的数学框架转化为构建软件的实用工具。你不再需要编写与代码逐渐脱节的需求文档，而是将系统描述为一个形式语境 K = (G, M, I)——对象、属性及其有向关联——然后让概念格机械地推导出接口契约、数据流、执行顺序和模块边界。

然后让 LLM 填充实现。

> **核心理念：** 把要构建的东西说清楚，*就是*在构建它。K 不是先于代码的规格说明——K 是代码所栖居的类型系统。

---

## 它做什么

从一份需求文档开始。KonceptOS 提取一个粗粒度的形式语境 K₀（例如 22 个功能 × 9 个关注点维度）。然后通过拆分对象和属性为更精细的词汇来**演化** K，将 RW（读写）歧义消解为精确的 R 或 W 方向。每一步都重算概念格 B(K)，揭示哪些模块必须就哪些数据契约达成一致。

当 K 足够精确时，运行 `build`，LLM 从 spec 生成完整的可运行应用。

```
requirements.md → llm analyze → K₀（粗粒度）
                                  ↓
                          resolve / evolve → K（精化）
                                  ↓
                              build → 可运行的应用
```

### 概念格提供什么

- **接口契约。** 每个概念 (A, B) 意味着：A 中的所有模块必须就 B 中每个通道的数据格式达成一致。
- **数据流图。** 每个通道上的写者和读者定义了完整的数据流，从 I 中机械推导。
- **执行顺序。** 写者必须先于读者执行。偏序关系从 I 中自然导出；偏序中的自由度就是可并行的部分。
- **一致性检查。** 某个通道有读者但没有写者？有写者但没有读者？自动检测。
- **影响分析。** 修改 K，运行 `diff`，获得数学上穷尽的受影响模块列表——更关键的是，**可证明不受影响**的模块列表。

### 有向关联

与经典 FCA（布尔值 I）不同，本系统使用有向值：

| 值 | 含义 |
|----|------|
| `0`  | 不涉及 |
| `R`  | 从该通道读取 |
| `W`  | 向该通道写入 |
| `RW` | 既读又写——说明通道定义还不够细 |

RW 不是系统的性质，而是**有损压缩的标志**。通过拆分对象或属性为更精细的词汇来消解 RW，是核心演化循环。

### 为什么这很重要——与传统开发对比

传统开发中，依赖关系是隐式的。它们散布在 import 语句、函数调用、全局变量引用和未文档化的约定中。修改一个接口后，影响范围靠人脑推断——遗漏是常态。

在 KonceptOS 中，每个依赖都是 I 中的显式条目。修改 K 后，`diff` 机械地、穷尽地列出：哪些关联值变了、哪些概念溶解了（旧契约不再需要）、哪些概念涌现了（新契约需要建立）、哪些模块需要重新实现。最重要的是，它列出**可证明不受影响**的模块——它们的 I 值没有改变，这是数学事实，不是主观判断。

传统开发只能说"我觉得这些模块没受影响。"K 说"这些模块的 I 值未变。"这个差异在规模化时最为关键：一个 200 模块的系统中，能证明 180 个模块不受某次修改的影响——这样的系统可以维护。一个 200 模块的系统中，必须人工逐一验证全部 200 个——这样的系统会腐烂。

### 自动一致性检查

K 在结构性错误被引入的那一刻就捕获它——不是在运行时，不是在代码审查中，不是在生产环境。

**有读者无写者（R_no_W）。** 某个模块从一个通道读取，但没有任何模块向该通道写入。数据从哪来？这是缺失的依赖——查看 `flows` 时立即检测到。

**有写者无读者（W_no_R）。** 某个模块向一个通道写入，但没有任何模块读取它。死输出——要么缺少消费模块，要么该通道是多余的。

**同一通道多个写者。** 两个模块都向同一个原子通道写入。这是潜在的竞态条件——也是该通道尚未原子化、需要继续拆分的信号。

**RW 作为一致性信号。** 每个 RW 格子都是一面旗帜：此处的词汇还不够精确。剩余 RW 格子的计数（显示在提示符 `K[|G||RW]` 中）是系统规约完备程度的实时进度指标。

这些检查不是贴在成品代码上的 linter。它们作用在架构本身，在任何代码写出之前。在 K 中捕获的问题修复成本是分钟级。同样的问题在运行中的系统里被发现，修复成本是天级。

### 与当前 AI 编程范式的区别

当前主流的 AI 编程工具遵循"生成 → 测试 → 修补"循环：用自然语言描述需求，让 LLM 生成代码，运行，发现 bug，让 LLM 修复。这对小程序有效。但随着系统增长会崩溃，因为：

**没有结构记忆。** 每轮生成都从零开始，或从上下文窗口中的先前对话出发。LLM 没有哪些模块依赖哪些数据的形式化模型——它每次都从代码中重新推断这些关系，而且每次推断的结果不同。

**架构静默漂移。** 随着 LLM 修补 bug 和增加功能，隐式架构以无人追踪的方式发生偏移。模块 A 开始依赖模块 B 的一个未文档化的副作用。当系统达到 50 个模块时，依赖图已不可知——一坨无法维护的屎山。

**工作流必须预设。** 大多数 AI 编程框架要求你在开始前设定显式的流水线、代理角色或多步工作流。开发过程的结构是预先决定并强加于问题之上的。

KonceptOS 采取不同的路径：

**K 是结构记忆。** 交叉表在所有交互中持久存在。每个模块的依赖是 I 中的显式条目，不是从代码中推断的。当 LLM 生成代码时，它在 K 定义的类型约束内工作——无法静默引入未声明的依赖。

**架构是一等公民。** 你演化 K（架构），代码跟随。传统 AI 编程演化代码，然后祈祷架构保持一致。当 K 变化时，`diff` 精确告诉你什么必须跟着变——什么可证明不用变。

**结构自然涌现，而非预先规定。** 你不需要预定义代理工作流或开发流水线。从需求文档出发，提取 K₀，演化它。概念格揭示系统的自然模块边界、接口契约和执行顺序。架构从领域中涌现，而非来自框架模板。

**一致性持续强制执行。** 每次修改 K 并运行 `compute`，概念格都会重算。结构性违规（缺失写者、孤儿读者、通道冲突）立即浮现。在 200 模块的系统中，这是可控增长与熵增之间的区别。

这使 KonceptOS 特别适合大型项目——在这些项目中，主要挑战不是写代码（LLM 已经擅长这个），而是在系统演化过程中维持架构一致性。

## 快速开始

```bash
python konceptos.py
```

进入交互式 REPL：

```
K[0|0]> llm analyze supermario_fca_spec.md
  22 obj, 9 attr
  K: |G|=22 |M|=9 RW=47 ?=0 |B|=52

K[22|47]> ctx          # 查看交叉表
K[22|47]> rw           # 列出所有 RW 格子
K[22|47]> resolve obj F13   # 将"方块系统"拆为子类型
K[25|50]> resolve attr D    # 将"游戏状态"拆为子通道
K[28|42]> compute      # 重算概念格
K[28|42]> build mario.html  # 生成完整应用
```

提示符 `K[|G| | RW]` 实时显示当前对象数量和剩余 RW 数量。

### 前置条件

- Python 3.8+
- [OpenRouter](https://openrouter.ai/) API key（或修改 `LLM` 类以接入你偏好的提供商）

```bash
export OPENROUTER_API_KEY="sk-or-v1-your-key-here"
python konceptos.py
```

## 种子

**种子**编码了领域知识，指导 K 如何演化——走哪条分解路径、用什么名字、期望什么关联方向。

种子有三个层级：

**Level 1 — 词表。** 一组合法的名字。LLM 消解对象或属性时只能从词表中选，保证命名一致性。

**Level 2 — 分解树 + 提示。** 对象和属性如何拆分的树结构，加上方向提示（例如 `"*|collider": "R"`——碰撞体永远是只读的）。这是最佳平衡点：足够的结构来引导演化，又足够通用以适用于同领域的不同项目。

**Level 3 — 参考 K\*。** 从已有实现中提取的完整原子级交叉表。新项目可以从 K\* 出发删改。

### 种子生命周期

```
t=0    领域专家提供 Level 1（词表）
       → 第一个项目硬做
t=1    从第一个项目的 K 快照链提取分解决策 → Level 2 种子
t=5    积累 5 个项目的 K* → 聚合为 Level 3 种子
t=20   种子稳定 → 发布
```

种子从使用中积累，不是从设计中创造。

### 示例种子

仓库包含 `seed_2d_platformer.json`——一个面向 2D 平台游戏的 Level 2 种子，包含：

- 对象词表（game_loop, physics_engine, patrol_enemy, ...）
- 属性词表（position, velocity, collider, sprite, ...）
- 分解树（entity → player, enemy, collectible, ...）
- 方向提示（renderer 永远读取 position，永远不写入它）
- **约定（Conventions）** — 跨通道值约束，例如跳跃可达性公式

约定在 `build` 时注入 LLM 提示，确保生成的代码遵守领域物理（例如平台在给定重力和跳跃速度常数下确实可以跳到）。

## 示例

仓库包含两个完整的工作示例：

### 超级马里奥（`supermario_fca_spec.md`）

完整的 FCA 驱动 spec，用于超级马里奥克隆。从 22 个功能 × 9 个关注点维度出发，通过对象和属性拆分演化到 28 × 16。概念格揭示了涌现的架构洞察——"道具流水线"（方块和道具共享 7/9 属性）、输入瓶颈（22 个功能中只有 2 个涉及用户输入）、以及"心脏簇"（方块、道具、旗杆是耦合度最高的节点）。

构建产物：`mario_v1.html`——一个可玩的超级马里奥游戏，单个 HTML 文件。

### 森林冰火人（`fireboy_spec.md`）

双人合作平台解谜游戏。展示了同一个种子（`seed_2d_platformer.json`）如何适用于结构上不同的游戏——不同的机制（元素免疫、开关、可推动箱子），但相同的底层关注点维度。

构建产物：`fireboy_v4.html`

## 理论简述

KonceptOS 建立在一个简单的观察之上：**一个软件系统的完整描述就是一张交叉表。**

```
K = (G, M, I)

G = 对象集（模块/函数）
M = 属性集（数据通道/关注点维度）
I : G × M → {0, R, W, RW}
```

从这张表计算出的概念格 B(K) 是系统的**接口契约层级结构**——从最粗的层级（"大家都是同一个系统"）到最细的层级（"这两个函数必须就这个数据通道的格式达成一致"）。

K 通过**词汇替换**从粗粒度的 K₀ 演化到精细的 K\*：用精确的名字替换粗名字，重填 I，重算 B(K)。旧概念可能**溶解**（被揭示为不精确词汇的幻影）。新概念可能**涌现**（更细的词汇暴露了之前不可见的契约）。

三种力驱动演化：

1. **自上而下的精细化。** 领域知识或种子提供分解规则。
2. **自下而上的反馈。** 实现揭示缺失或错误的契约。
3. **横向发现。** 构建一个模块时发现需要一个新的数据通道。

三种力交替发生。K 的演化与编码不是先后关系——它们交织在一起。

详细内容参见 `TUTORIAL中文版.md`。
