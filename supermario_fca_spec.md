# Super Mario FCA 需求规格 — 概念格驱动的 Webapp 重构文档

> **目的**：本文档用形式概念分析 (FCA) 替代传统需求列表。
> 请读完整份文档后，实现一个可运行的 2D 横版平台跳跃 Webapp（浏览器端，HTML+JS+Canvas），
> 功能范围严格由概念格定义——**格中有的做，格中没有的不做**。

---

## §0 阅读指南

本文档的结构不是线性的"功能清单"，而是一个**偏序格 (Lattice)**。

- **§1 形式背景** 是原始数据——22 个功能 × 9 个属性的交叉表。
- **§2 概念格** 是从交叉表中通过闭包运算自动计算出的所有形式概念，按层次排列。
- **§3 实现映射** 将每个概念节点映射到你需要写的代码模块。
- **§4 约束与边界** 说明什么做、什么不做。

**核心原则**：每个"形式概念"是一组功能（外延）和一组属性（内涵）的最大匹配。
你实现的代码模块应当与概念节点一一对应——模块边界来自数学，而非主观判断。

## §2 概念格 𝔅(𝕂) — 自动涌现的需求层次

以下概念由闭包运算 (·)'' 从形式背景中计算得出。
层次由 Hasse 图的拓扑排序决定——**不是人为划分**。

### L0 — 顶概念 ⊤（最大泛化）

**C00** `({全部22个功能}, ∅)`
- 内涵：空集（无公共属性）
- 含义：**"这是一个超级马里奥游戏"** — 所有功能的最抽象共性
- 实现：→ 你的 `main()` 或 `App` 入口

### L1 — 核心正交轴（单属性概念）

每个概念对应一个独立的关注点维度。L1 是系统的"骨架"。

**C01** `({F01,F05,F06,F11,F13,F14,F15,F16,F17,F18,F20,F22}, {A:图形渲染})`
- 所有向屏幕绘制的功能（12个）
- 实现：→ `Renderer` 模块（Canvas 2D 上下文、精灵绘制、清屏）

**C02** `({F04,F06,F08,F09,F10,F11,F12,F13,F14,F15,F16,F17,F18}, {B:空间物理})`
- 所有涉及位置、速度或 AABB 的功能（13个——覆盖面最广！）
- 实现：→ `Physics` 模块（Vec2、AABB、重力常量、速度积分）

**C03** `({F03,F04}, {C:用户输入})`
- 响应键盘的功能（仅2个——高度内聚！）
- 实现：→ `Input` 模块（keydown/keyup 状态表、按键映射）

**C04** `({F05,F07,F12,F13,F14,F18,F19,F20}, {D:游戏状态})`
- 读写分数/金币/生命/时间/变身状态的功能
- 实现：→ `GameState` 对象（score, coins, lives, time, marioBody）

**C05** `({F06,F11,F12,F13,F14,F15,F16,F17,F19}, {E:实体生命周期})`
- 运行时动态创建或销毁对象的功能
- 实现：→ `EntityManager`（怪物、道具、粒子、火球等类型化数组）

**C06** `({F02,F03,F04,F05,F06,F08,F09,F10,F11,F13,F14,F15,F16,F17,F18,F20}, {F:帧驱动})`
- 每帧/每 tick 被调用的功能（16个——最广的横切关注点！）
- 实现：→ `GameLoop`（requestAnimationFrame + 100Hz 固定步长）

**C07** `({F04,F05,F06,F07,F09,F12,F13,F14,F15,F18,F19,F21}, {G:音频反馈})`
- 触发声音的功能（12个）
- 实现：→ `AudioManager`（Web Audio API / HTMLAudioElement 池、BGM 控制）

**C08** `({F04,F05,F06,F11,F13,F14,F17,F18,F22}, {H:动画})`
- 有精灵帧循环或视觉变换的功能
- 实现：→ `Animator`（帧计数器模式：effF/effCnt/maxFrame/speed）

**C09** `({F07,F15,F18,F19,F21,F22}, {I:关卡结构})`
- 涉及关卡数据、场景切换、关卡类型分支的功能
- 实现：→ `LevelManager`（关卡定义、加载/销毁、类型→资源映射）

### L2 — 二元组合层（两属性交叉）

这些概念揭示了**哪些关注点天然耦合**——即模块接口的边界。

**C10** `({F06,F11,F13,F14,F15,F16,F17}, {A渲染 ∧ B物理})`
- 既绘制又有物理的东西——**游戏世界实体**
- 含义：可绘制的物理对象需要统一的实体接口
- 实现：→ `Entity` 基类（x, y, vx, vy, width, height, draw(), update()）

**C11** `({F06,F11,F13,F14,F17}, {A渲染 ∧ B物理 ∧ E生命周期})`
- 可绘制、有物理、且动态创建/销毁的对象
- 含义：**动态实体池**——运行时生成和回收

**C12** `({F04}, {B物理 ∧ C输入 ∧ F帧驱 ∧ G音频 ∧ H动画})`
- 马里奥移动是**唯一同时结合输入+物理+音频+动画**的功能
- 含义：马里奥控制器是系统中最复杂的单功能节点
- 实现：→ `MarioController`（读取 Input → 更新 Physics → 触发 Audio → 驱动 Animation）

**C13** `({F05,F13,F14,F18}, {A渲染 ∧ D状态 ∧ F帧驱 ∧ G音频 ∧ H动画})`
- 渲染+游戏状态变更+反馈的组合
- 含义：**奖励/变身反馈环**——视觉+音频+分数变化
- 实现：→ "事件发生→状态改变→播放声音→播放动画"的统一模式

**C14** `({F13,F14}, {A渲染 ∧ B物理 ∧ D状态 ∧ E生命 ∧ F帧驱 ∧ G音频 ∧ H动画} — 7个属性！)`
- 方块和道具共享 **9个属性中的7个**——系统中最高重叠度的一对
- 含义：方块和道具在架构上几乎相同。顶方块→生成道具→拾取道具→改变状态。它们构成**道具管线**
- 实现：→ `BlockManager` + `PropManager`，共享事件接口

**C15** `({F15,F18}, {A渲染 ∧ B物理 ∧ F帧驱 ∧ G音频 ∧ I关卡})`
- 管道传送和旗杆过关都涉及渲染+物理+音频+关卡切换
- 含义：**关卡跳转桥梁**——离开当前关卡的两种方式
- 实现：→ 共享 `LevelTransition` 处理器

**C16** `({F07,F19}, {D状态 ∧ G音频 ∧ I关卡})`
- 死亡重生和场景管理都改变状态+播放音频+重建关卡
- 含义：**生命周期重置事件**——都触发场景重建
- 实现：→ `SceneReloader`（销毁实体→重载关卡→重置状态）

**C17** `({F03,F04}, {C输入 ∧ F帧驱})`
- 输入轮询和马里奥移动是**仅有的两个响应输入的功能**
- 含义：输入通过极窄的管道汇入唯一消费者（马里奥）
- 实现：→ `Input.poll()` → `MarioController.handleInput()`

**C18** `({F08,F10}, {B物理 ∧ F帧驱})`
- 物理和摄像机是纯空间+帧驱动功能，不渲染也不改状态
- 含义：**被动空间基础设施**——只计算，不显示
- 实现：→ 薄工具模块

**C19** `({F06,F11,F17}, {A渲染 ∧ B物理 ∧ E生命 ∧ F帧驱 ∧ H动画})`
- 火球、怪物、粒子是三种**有动画、动态、有物理、可渲染**的实体
- 含义：**自治世界对象**——独立存活、移动、动画、死亡
- 实现：→ 都继承 `Entity` 基类 + 生命周期 + 动画混入

### L3 — 三元及以上融合层（高耦合功能）

**C20** `({F13}, {A渲染, B物理, D状态, E生命, F帧驱, G音频, H动画} — 7个属性)`
- 方块系统横跨 **9个维度中的7个**——系统中耦合度最高的节点之一
- 它渲染、有空间顶起动画、改变游戏状态（金币）、创建道具、
  每帧运行、播放声音、有?块闪烁动画
- 含义：**架构心脏**——方块是大多数子系统的交汇点
- 实现：→ `BlockManager` 必须精心设计，与7个系统保持清晰接口

**C21** `({F14}, {A渲染, B物理, D状态, E生命, F帧驱, G音频, H动画} — 7个属性)`
- 道具系统：与方块同样复杂，道具管线的另一端
- 实现：→ `PropManager`，通过生成事件与 BlockManager 紧密耦合

**C22** `({F18}, {A渲染, B物理, D状态, F帧驱, G音频, H动画, I关卡} — 7个属性)`
- 旗杆过关也横跨7个维度——**胜利序列**
- 它渲染旗帜、有物理（旗帜下滑）、改变状态（时间→分数）、
  每帧运行、播放胜利音乐、有马里奥攀爬动画、触发关卡切换
- 实现：→ `VictorySequence` 状态机（抓旗→下滑→行走→进城堡→下一关）

**C23** `({F06}, {A渲染, B物理, E生命, F帧驱, G音频, H动画} — 6个属性)`
- 火球：可渲染、有物理、动态创建/销毁、每帧运行、播放声音、有旋转动画
- 实现：→ `FireballManager`，由马里奥生成，碰墙/碰怪后销毁

**C24** `({F15}, {A渲染, B物理, E生命, F帧驱, G音频, I关卡} — 6个属性)`
- 管道：可渲染、有空间位置、生命周期管理、每帧运行、播放声音、触发关卡切换
- 实现：→ `PipeManager`，进入子关卡的主要机制

## §4 约束与边界

### 4.1 必须实现（概念格中存在的）

1. 基于 Canvas 的 2D 横版过关游戏，可在浏览器中运行
2. 马里奥 3 种形态：小人 (child)、大人 (adult)、火人 (old/fire)
3. 移动：左右奔跑带加速+摩擦，跳跃带可变高度，蹲下
4. 至少 2 种怪物：板栗仔（踩死）和乌龟（踩→缩壳→踢壳→壳滑行并杀敌）
5. ?块从下方顶击后弹出道具；砖块被大马里奥顶碎
6. 道具：蘑菇（小→大）、太阳花（大→火）、星星（无敌）、金币（+1金币）
7. 管道进入（站在管道上按下蹲键）→ 切换到子关卡 → 返回管道
8. 碰撞：马里奥 vs 地面/墙壁，马里奥 vs 怪物（顶部=踩踏，侧面=受伤），马里奥 vs 道具
9. 摄像机向右跟随马里奥，永不向左回退
10. 关卡末尾的旗杆：抓住→旗帜下滑→马里奥走进城堡→剩余时间转化为分数
11. 粒子效果：砖块碎裂时碎片飞溅、得分文字向上浮动
12. HUD：分数、金币数、时间倒计时、生命数、关卡名
13. 音频：每种关卡类型对应 BGM，音效包括跳跃、金币、踩踏、死亡、变身、砖块碎裂
14. 至少一个完整关卡：地面砖块、缺口、管道、方块、怪物、旗杆
15. 生命系统：掉入深渊或小人状态被怪物碰到→扣命→检查点重生
16. 生命为 0 时 Game Over

### 4.2 不要实现（概念格中不存在的）

- 无多人模式
- 无世界地图/关卡选择画面
- 无水下关卡/游泳机制
- 无 Boss 战（库巴等）
- 无传送区域（管道只通往预定义子关卡）
- 无存档/读档系统
- 无复杂菜单系统（按任意键开始即可）
- 无高级敌人（刺猬龟、子弹比尔、锤子兄弟等）
- 无藤蔓攀爬机制
- 无 100 金币换命机制（可简化忽略）

### 4.3 技术选型约束

- 单文件 HTML（或 HTML + 内联 JS），浏览器直接打开
- 仅使用 Canvas 2D API——不用 WebGL，不依赖框架
- 精灵图以内嵌 base64 加载，或用彩色矩形程序化绘制（占位图形可接受）
- 瓦片式关卡：16×16 像素瓦片，关卡定义为二维数组或字符串地图
- 目标 100 FPS 固定步长（匹配原始源码）
- 键盘输入：方向键或 WASD 移动，Z/空格跳跃，X 奔跑/射击

## §6 涌现的架构洞察

### 6.1 "心脏"集群

FCA 揭示 **方块 (F13)、道具 (F14)、旗杆 (F18)** 各横跨 9 个属性中的 7 个——系统中最高耦合度。它们是渲染、物理、状态、生命周期、帧更新、音频、动画共同交汇的点。传统文档会把它们列为三个独立功能；概念格表明它们是**相同架构模式的三个实例**。如果你把方块、道具和旗杆设计为相同接口，代码量减半。如果三者之一有 bug，bug 大概率涉及相同的 7 个子系统。

### 6.2 输入瓶颈

22 个功能中仅 **2 个**涉及用户输入（F03, F04）。这是 2D 平台游戏的签名特征：输入面极小，但通过物理、音频和动画扇出到 13+ 个下游功能。C03 是**最窄的 L1 概念**。

### 6.3 帧驱动主导

{F:帧驱动} 覆盖 **22 个功能中的 16 个**——最广的横切关注点。这意味着：几乎所有东西都每帧运行。不在帧循环中的例外（F01 窗口初始化、F07 死亡、F12 踩踏、F19 场景管理、F21 音频、F22 精灵）是事件触发或一次性操作。这个区分在架构上很重要：帧驱动功能进游戏循环；事件驱动功能进事件分发器。

### 6.4 音频扩散

音频（属性 G）出现在 **22 个功能中的 12 个**——几乎和帧驱动一样广。但与帧驱动不同，音频始终是**副作用**，从非主要关注点。概念格确认：除 F21 外没有功能*仅*为音频而存在。这意味着音频应当是**即发即忘的旁路通道**，不是核心循环的参与者。

### 6.5 道具管线的涌现

C14（L2 层的 `{F13,F14}`，共享 7 个属性）不是任何人会写成需求条目的东西。它是从数学中**涌现**的桥梁概念——"顶方块→生成道具→拾取道具→改变状态"是一条管线，不是两个独立功能。
